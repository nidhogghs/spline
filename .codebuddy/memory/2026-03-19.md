# 2026-03-19 工作日志

## 重写 main_1.py 为真正的增量算法

### 背景
用户发现 main_1.py 虽然有 O/C/N 分区逻辑，但实际上 `extend_one_stage` 仍然把所有历史数据 append 到 `t_all/X_all/y_all`，在全部数据上构造设计矩阵、做优化和 CV，是"伪增量"。

### 核心改动
1. **去除全局数据累积**：不再在 trainer 中维护 `t_all/X_all/y_all`
2. **extend_one_stage 只接收 CN 区间数据**：参数从 `(t_seg, X_seg, y_seg)` 变为 `(t_cn, X_cn, y_cn)`
3. **新增 `collect_cn_data()` 函数**：从上一段数据的尾部（覆盖 C basis 的旧侧 support）+ 当前段的新采样数据组合出 CN 区间数据
4. **cv_select_lambda_frozen_cn**：新的 CV 函数，只在 CN 残差数据上做交叉验证（不再像旧版在全部数据上构造 B_full）
5. **RMSE 只在 CN 区间上计算**：输出字段改为 `train_rmse_cn`
6. **checkpoint 不再保存数据**：只保存 knots + coef_mat（真增量不需要数据缓存）
7. **run_or_resume_incremental 维护 prev_seg 数据**：只保留上一段的采样数据，用于下一阶段的 C basis 数据覆盖

### Smoke Test 结果 (t_final=4, P=80, n=80)
- Stage 2: n_cn_data=95, range=[0.735, 1.992], rmse_cn=0.034
- Stage 3: n_cn_data=110, range=[1.715, 2.988], rmse_cn=0.021
- Stage 4: n_cn_data=101, range=[2.718, 3.998], rmse_cn=0.020
- 所有 OCN 验证通过

## 锯齿问题分析与修复

### 问题描述
用户反映 beta(t) 估计曲线在 stage 交界处（整数 t 附近）有明显锯齿/振幅偏差。

### 分析过程
1. **对比 git 历史**：加速提交 (413be1f) 将 FISTA 从显式 Gram 矩阵 (Φ^TΦ) 改为隐式矩阵向量乘法 `fista_group_lasso_design`
2. **尝试的修复方案**：
   - `gram_R` 积分范围从 CN 局部改为全局 [0, next_end] ✓（保留）
   - Lipschitz 估计改进：迭代次数 30→80，余量 2%→10%，随机初始化 ✓（保留）
   - Warm start: 用旧 C 系数初始化 FISTA ✓（保留）
   - C basis 降权 (0.25): ✗ 导致过拟合
   - 完全冻结 C basis: ✗ 因为 C basis 在新旧 knots 中形状不同
3. **根因分析**：锯齿本质是真增量算法在 n_per_segment=100, P=100 设置下 CN 数据不足（~130 点 vs 1300 参数），而非加速改动的数值 bug。原始"伪增量"版本用全量数据（stage 10 有 1000 点），所以锯齿不明显。

### 最终保留的改进
1. `gram_R` 在全局 [0, next_end] 上积分（数学上更正确）
2. `_estimate_lipschitz_design`: 80 次迭代 + 10% 余量 + 随机初始化
3. FISTA warm start: C basis 用旧系数初始化
4. 速度：从 310s 降至 96s（20 stages）

## 恢复原始 FISTA 算法质量（保持增量特性）

### 背景
用户认为之前的"加速计算"改动（隐式矩阵乘法 + 功率迭代）是锯齿的根源，要求参考 ipynb 原始算法恢复质量。

### 核心修改
1. **FISTA 恢复为显式 Gram 矩阵版**：`fista_group_lasso(XtX, Xty, ...)` + `np.linalg.norm(XtX, 2)` 精确谱范数
   - 删除 `fista_group_lasso_design`、`_estimate_lipschitz_design`、`_apply_smooth_gram`
2. **group_soft_thresh 恢复逐组循环**：`for v, wp in zip(blocks, w): v.T @ R @ v`
3. **恢复 adaptive_cn_refit**：与 main.py 一致的自适应 group-lasso 二次拟合
4. **恢复 r_relax=2**：将最靠近边界的 2 个 O basis 释放加入 CN 优化
5. **CV 恢复显式 Gram 构造**：每个 fold 内构造 `Phi_tr.T @ Phi_tr`
6. **lambda_max_R 恢复 VCM 设计矩阵版**：`lambda_max_R(Phi, y, m, R)`
7. **保持增量特性不变**：不累积全量数据，只用 CN 区间数据

### 结果 (configs/main1_t20_p100_n300_simple5_v2.json)
- Beta RMSE: β₁=0.0352, β₂=0.0368, β₃=0.0321, β₄=0.0381, β₅=0.0361
- 锯齿显著改善
- **稀疏性大幅改善**：Active 从 100/100 降至 21/100（adaptive refit 的效果）
- 耗时约 7036s（~117 分钟），每 stage ~6 分钟（因显式 Gram + adaptive 增加计算量）
- CN basis 数从 13 增至 15（r_relax=2 释放了 2 个边界 basis）

## 锯齿根因最终定位：r_relax=2 破坏旧区间

### 诊断结论
- **根因**：`r_relax=2` + 只用 CN 数据的组合导致被释放的 basis（support 深入旧区间，如 [0.5, 0.9]）在重新拟合时缺乏旧区间数据约束，系数偏移高达 0.25，破坏旧区间 β(t) 估计
- **证据**：r_relax=0 时 MISE=0.00080，r_relax=2 时 MISE=0.00161（翻倍），r_relax=3 时 MISE=0.027（灾难）
- adaptive refit 在平均意义上也恶化结果（p=5 MISE 增加 58%）
- main.ipynb 不受影响是因为用全量数据约束 CN basis

### 修复方案（已实施）
1. `r_relax` 默认值从 2 改为 0（完全冻结 O basis）
2. 新增 `adaptive` 参数，默认 False（不执行 adaptive refit）
3. IncrementalVCMTrainer 的 __init__、save_checkpoint、load_checkpoint 均已更新

### 修复后结果 (P=15, n=300, t_final=8)
- 增量 vs Batch 的 MISE 比值：p=1: 1.05x, p=2: 1.08x, p=3: 1.03x, p=4: 1.01x, p=5: 1.07x, 平均: 1.05x
- **增量方法几乎与 Batch 全局拟合一样好！** 锯齿完全消失
