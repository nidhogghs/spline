## beta_1 添加 log 尾部偏移
- 修改 `true_beta_funcs_default()` 中 β₁ 的定义
- 原始: `scales[0] * (-0.5 + 0.6 * cos(2πt))`
- 修改后: `scales[0] * (-0.5 + 0.6 * cos(2πt)) + 0.15 * log(1 + 0.3t)`
- log 项使 β₁ 不再是纯周期函数，具有单调递增偏移，尾部值逐渐升高

## 尖峰低估分析
### 现象
- 所有变量在极值点处均存在系统性低估（shrinkage ratio < 1）
- Mean shrinkage: p=2: 0.9665, p=3: 0.9586, p=4: 0.9568, p=5: 0.9531
- **增量和 Batch 的低估程度几乎一样**（差距仅 0.1-0.2%），说明这不是增量算法的问题

### 根本原因：Group-Lasso 的固有收缩偏差 (Shrinkage Bias)
1. **L1/Group-Lasso 惩罚的本质**：Group-Lasso 对系数施加向零收缩的惩罚，使得估计值在幅度上系统性地小于真实值
2. **尖峰处最严重**：在极值点处，真实值的绝对值最大，收缩效应最明显
3. **1SE 规则加剧**：1SE 选择的 lambda 比 MinCV 大 1.30 倍（0.878 vs 0.675），导致更强的收缩
4. **实验验证**：MinCV 的 MISE 比 1SE 低约 6-17%（p=3: 0.00074 vs 0.00086）

### 解决方案建议
1. **改用 MinCV** 代替 1SE（但牺牲稳定性）
2. **增加样本量** n_per_segment：更多数据 → 更小的 lambda → 更少收缩
3. **后处理 debiasing**：拟合后对活跃组做无惩罚重新拟合
4. **减小 knot_step**：更细的 knot 间距可以更好捕捉尖峰

## Post-Lasso Debiasing 实验（方案1）
- 在 `main_1.py` 中实现了 `post_lasso_debias()` 和 `post_lasso_debias_cn()` 函数
- 思路：Group-Lasso 选出活跃组 → 对活跃组做无惩罚 OLS 重拟合
- 增量版：Stage 1 全局 debias; Stage s>1 只对 CN 子空间做 debias

### 实验结果（与之前 setting 完全一致: P=15, n=300, t=8, sigma=0.1）
- **尖峰 shrinkage 改善显著**: 平均从 0.967 → 0.993（几乎完美！）
  - p=3: 0.959 → 0.991, p=4: 0.957 → 0.996, p=5: 0.953 → 0.998
- **但 MISE 大幅恶化**: 平均从 0.000725 → 0.007273（恶化 ~10x）
  - 边界处（t≈0, t≈8）出现巨大误差尖峰
  - 高频函数 sin(4πt) 受影响最大（MISE 从 0.0009 → 0.012~0.016）

### 原因分析
- Post-Lasso Debiasing 成功消除了收缩偏差（bias ↓），但引入了更大的方差（variance ↑）
- 经典的 **bias-variance tradeoff**：OLS 在高维 B-spline basis 上方差很大
- 边界效应：B-spline 在端点附近的 basis 稀疏，OLS 不稳定导致边界爆炸
- **结论**：在当前 setting 下，朴素的 Post-Lasso Debiasing 不可取；Group-Lasso 的轻微 shrinkage bias 远小于 debias 引入的方差代价

## MinCV vs 1SE 实验（方案2）
- 脚本：`_run_mincv_vs_1se.py`
- 对比 4 种方法：Incr 1SE / Incr MinCV / Batch 1SE / Batch MinCV

### 实验结果（P=15, n=300, t=8, sigma=0.1）
- **MISE 全面改善**：MinCV 平均 MISE 从 0.000725 → 0.000596（改善 **17.7%**）
  - p=1: 16.0%, p=2: 5.8%, p=3: 22.9%, p=4: 29.4%, p=5: 12.1%
  - 高频函数 sin(4πt) 受益最大（p=4 改善 29.4%）
- **尖峰 shrinkage 明显改善**：平均从 0.967 → 0.976
  - p=3: 0.959 → 0.972, p=4: 0.957 → 0.969, p=5: 0.953 → 0.969
- **lambda 对比**：MinCV 选择的 lambda 比 1SE 小约 30-40%
  - Stage 1: 0.141 vs 0.239; Batch: 0.675 vs 0.878
- **无副作用**：没有边界爆炸或方差增大的问题
- **结论**：MinCV 是一个简单有效的改善方案，在 MISE 和 shrinkage 上都优于 1SE，且没有任何负面代价

## 密节点 + 大样本量实验
- 脚本：`_run_dense_knots_large_n.py`
- 在 MinCV 基础上测试 knot_step 和 n_per_seg 的影响
- 对比 6 种配置：原始(ks=0.1,n=300) / 密节点(ks=0.05,n=300) / 大n(ks=0.1,n=600) / 两者兼(ks=0.05,n=600) / Batch / 1SE基线

### 关键发现
- **增大n是主要改善因素**：ks=0.1, n=600 改善 **66.9%**（MISE: 0.000725→0.000240），是最佳增量配置
- **仅增密节点反而恶化**：ks=0.05, n=300 MISE 恶化 **-123.5%**（参数量翻倍但数据不够，过拟合）
- **密节点+大n**：ks=0.05, n=600 改善 **36.7%**（MISE: 0.000458），不如仅增大n
- **增量≈Batch**：ks=0.05, n=600 增量(0.000458)≈Batch(0.000461)，再次验证增量算法质量
- **Shrinkage**：n=600 将 shrinkage 从 0.967 → 0.988（接近完美），远优于 MinCV 单独改善
- **结论**：增加样本量(n)远比增加节点密度(knot_step)重要；密节点在数据不足时会恶化结果
